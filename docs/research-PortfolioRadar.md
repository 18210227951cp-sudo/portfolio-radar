# 第一步：市场调研与产品定位分析（研究阶段）

## 1.1 目标用户画像

### 典型用户 1：入门/轻度投资者（小白）
- 特征：持仓 1–10 只，主要是指数基金/大盘 ETF/少量热门股；关注“我到底赚没赚、赚多少”。
- 他们当前用什么方式管理持仓：
  - 记在备忘录/Excel/微信收藏；或只看券商 APP 的总资产。
- 痛点：
  - 不会做成本、仓位、收益率的标准化计算；
  - 持仓分散后难以快速看清“哪只拖累/贡献最多”；
  - 多平台买入（不同券商/基金平台）难统一。

### 典型用户 2：多标的配置型用户（理财/资产配置）
- 特征：持仓 10–50 只，股票+基金+ETF 混合；看重仓位结构、风险暴露与再平衡。
- 他们当前用什么方式管理持仓：
  - Excel/Notion 表格；或看平台“持仓分析”截图保存；偶尔用第三方记账/投顾应用。
- 痛点：
  - 手工维护成本高，更新频率低导致数据失真；
  - 缺少“组合视角”的统一指标（总成本/总市值/总盈亏/收益率）；
  - 图表不够直观或信息过载。

### 典型用户 3：短线/高频关注者（交易型）
- 特征：持仓变化频繁，关注涨跌幅排名、当日波动与单票贡献度。
- 他们当前用什么方式管理持仓：
  - 券商自选股 + 手动记成本；或在多个 APP 来回切换；部分人做自己的表格。
- 痛点：
  - 成本与仓位经常变动，容易算错；
  - 想要“按盈亏/涨跌排序”快速定位问题仓位；
  - 不需要复杂研究工具，只要快、清晰、可复盘。

## 1.2 竞品分析

### 竞品 1：雪球（组合/自选与持仓展示）
- 核心功能：自选跟踪、讨论社区、组合展示、部分数据可视化。
- 不足：
  - 信息流与内容干扰强，做“纯持仓管理”不够专注；
  - 自定义字段/成本口径不统一；
  - 用户想要更轻量、离线、可控的数据模型较难。

### 竞品 2：且慢（基金组合/投顾视角）
- 核心功能：组合配置、基金持仓结构、投顾策略与长期跟投体验。
- 不足：
  - 更偏基金/投顾体系，股票与多资产灵活性不足；
  - 组合分析口径受平台限制，用户“自建组合+自定义成本”空间有限；
  - 对只想快速看盈亏与仓位的用户来说偏重。

### 竞品 3：东方财富（持仓/资产分析页面）
- 核心功能：账户资产与持仓明细、盈亏展示、部分图表、交易闭环。
- 不足：
  - 强绑定账户体系，跨券商/多平台难整合；
  - UI 功能复杂，学习成本高；
  - 轻量导出、快速录入/编辑、统一视图不够“极简”。

## 1.3 产品定位

### 我们的差异化价值
- 轻量：打开即用，专注“组合视角”核心指标与结构图，不做内容社区、不做重研究。
- 可控：用户自己录入成本与数量，计算口径透明一致；数据本地保存，不依赖登录与云端。
- 快速洞察：一眼看懂总盈亏、收益率、仓位结构，并能按盈亏/涨跌快速排序定位问题持仓。
- 零环境依赖：纯前端单页（CDN 引入），方便放 GitHub Pages 直接分享与使用。

### MVP 聚焦功能（MoSCoW）

#### Must（必须有）
- 持仓录入（名称/代码/成本价/现价/数量）
- 自动计算（单票：市值、盈亏金额、盈亏比例；组合：总成本/总市值/总盈亏/总收益率）
- 仓位饼图（各持仓资金占比）
- 持仓列表展示（清晰的红亏绿盈格式化）

#### Should（应该有）
- 编辑/删除持仓（纠错与维护必需）
- 本地存储 localStorage（刷新不丢）
- 排序/排行榜（按盈亏/涨跌幅/市值等）

#### Could（可以有）
- 导出 CSV（用于备份与二次分析）
- 简单筛选/搜索（按代码/名称）

#### Won’t（本版不做）
- 实时行情 API 对接（数据源与频控复杂，后续版本再做）
- 账户登录/云端同步（引入后端与合规成本，MVP 不做）
